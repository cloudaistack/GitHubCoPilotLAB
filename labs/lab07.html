Lab 7: TDD with Copilot

Practice the Red-Green-Refactor cycle of Test-Driven Development with AI assistance. Write tests first using Copilot, then implement code to make them pass.

🎯 Objectives

  • Write comprehensive tests before any implementation code
  • Use Copilot to generate pytest tests from specifications
  • Implement functions to make failing tests pass
  • Use parametrized tests and fixtures
  • Cover edge cases systematically

📋 Prerequisites

  • Python 3.8+ with pytest installed (pip install pytest)
  • Completed Module 7

📝 Step-by-Step Instructions

1The Specification

You'll build a StringProcessor class with these methods. Do NOT implement them yet — you'll write tests first!

After each test comment, press Enter and let Copilot generate the test method. Continue for all five methods. Aim for at least 3-5 tests per method.

💡
Pro Tip: Use @pytest.mark.parametrize for data-driven tests. Write the first parametrize decorator and Copilot will suggest test data:
Example parametrized test
@pytest.mark.parametrize("input_text, expected", [
    ("racecar", True),
    ("A man a plan a canal Panama", True),
    ("hello", False),
    ("Was it a car or a cat I saw?", True),
    ("", True),
])
def test_is_palindrome(self, processor, input_text, expected):
    assert processor.is_palindrome(input_text) == expected
3Run Tests — Watch Them Fail (Red)
  1. Run: pytest test_string_processor.py -v
  2. All tests should FAIL (because methods return None)
  3. This confirms your tests are actually checking something
✅ Expected Output
All tests should show FAILED with assertion errors. This is the "Red" phase of TDD — failing tests that define the desired behavior.
4Implement with Copilot (Green Phase)

Now go back to string_processor.py. For each method:

  1. Delete the pass statement
  2. Let Copilot suggest the implementation based on the docstring
  3. Accept the suggestion
  4. Run pytest -v to see if the tests pass
  5. If tests fail, use /fix to debug
ℹ️
Because your test file is open in another tab, Copilot uses it as context when generating implementations. It "knows" what behavior is expected from your tests!
5Refactor (Refactor Phase)
  1. Once all tests pass, select each method
  2. Use /refactor to improve the implementations
  3. Run tests after each refactor to ensure nothing breaks
  4. Add /doc to generate complete docstrings
